# 数据拉取与实时机制（看这一篇就不会搞混）

> 目的：把"数据怎么自动进库、比赛时钟怎么走、状态码什么意思、额度够不够、怎么部署"
> 一次性讲清楚，以后回来看不会混。

## 一、整体架构：写库 / 读库 彻底分离

```
API-Football v3  ──(poller 守护进程, 写)──►  data/wc.db  ──(Next.js, 只读)──►  前端页面
   外部数据源                scripts/poller.mjs        SQLite        lib/* + components/*
```

- **poller.mjs（写）**：常驻后台循环，按比赛状态自适应频率，把
  状态 / 比分 / 时间(分钟+补时) / 事件 / 赛前阵容 / 裁判 写进 `data/wc.db`。
- **Next.js 应用（读）**：所有页面只读这个库，从不在请求链路里调外部 API。
  因此前端高频轮询 `/api/live` 没有额度压力。
- 两者**解耦**：可各自独立启动 / 重启 / 部署。poller 挂了，前端仍按库里最后的数据正常显示。

> **SQLite 驱动**：原生模块 `better-sqlite3` 在本环境无法编译（无预编译包、无 C++ 工具链，
> 报 `Could not locate the bindings file`），已统一改用 Node 内置的 **`node:sqlite`**。
> 兼容层 `lib/db/sqlite-node.mjs` 实现了与 better-sqlite3 一致的接口
> （`prepare`/`exec`/`pragma`/`transaction`/`raw()` 等），**应用（drizzle 适配器）与全部抓取脚本共用**，
> 零原生依赖、无需编译。如遇 DB 加载报错，先查这里。

## 二、数据从 API 到前端的完整链路

| 环节 | 文件 | 作用 |
|------|------|------|
| 拉取写库 | `scripts/poller.mjs` | 按状态自适应频率写 `live_scores` 等表 |
| 库表定义 | `lib/db/schema.ts` `lib/db/client.ts` | `live_scores` 含 `status_short/elapsed/extra/kickoff_ms` |
| 读库 | `lib/db/live-scores.ts` | `getLiveScores()` → `LiveScoreEntry[]` |
| 组装 overlay | `lib/live-overlay.ts` | `getLiveOverlay()` → `LiveInfo`（含真实状态码/分钟/补时） |
| HTTP 端点 | `app/api/live/route.ts` | `/api/live` 返回 overlay（`no-store`，纯读库） |
| 客户端注入 | `components/live-provider.tsx` | `LiveProvider` 每 60s 拉 `/api/live`，仅内容变化时换引用 |
| 实时时钟 | `components/live-provider.tsx` 的 `LiveMinute` | 用真实状态机本地逐秒走 `mm:ss` |
| 类型 | `lib/data.ts` 的 `LiveOverlayInfo` | 客户端安全类型（不引入 server-only） |

**关键点**：`/api/live` 每 60s 拉一次（拿最新分钟/比分/状态），而**秒针在客户端本地走**，
不靠网络。所以即使 poller 是 20s 一轮、前端 60s 拉一次，用户看到的钟也是逐秒平滑跳动的。

## 三、比赛时钟怎么走（状态机 + 本地补秒）

### 真实分钟从哪来
API-Football 返回 `fixture.status.elapsed`（**只到分钟，没有秒**）和
`fixture.status.extra`（伤停补时分钟，如上半场 45+2 时 `extra=2`）。poller 把它们写进
`live_scores.elapsed / extra / status_short`。

### 秒怎么补（分钟与秒针同源，确定性、刷新不归零、进位同步）
前端 `LiveMinute` 把**分钟与秒针同时锚定到固定 `kickoffMs` 的整秒计数** `K(t)=floor((now−kickoffMs)/1000)`：
- `second = K(now) % 60` —— 当前时刻的**确定性函数**，任何设备/任何时刻刷新都得同一连续秒数，**poller 每 20s 重抓也不重置**。
- 分钟进位 = `K` 跨过的整分钟数，与秒针 `59→00` 翻转**严格同一刻**发生。

分钟基准仍以 API 的 `elapsed` 为权威（足球有中场/伤停，墙上时钟≠比赛分钟）：
`minute = baseMinute + (floor(K(now)/60) − floor(K(asOfMs)/60))`，中场时长在差值里自然抵消；
本地相对 `asOfMs` 多走的分钟封顶 `MAX_LOCAL_DRIFT_MIN`（2 分钟）防 poller 停摆跑飞。
实现见 `lib/data.ts` 的 `tickClock(clock, kickoffMs, asOfMs, nowMs)`。

> 历史 bug：① 旧版秒针锚 `asOfMs`，每次抓取/刷新秒数从 0 重算；② 后来分钟用 `asOf`、秒针用 `kickoff`
> **两套时钟脱节**，到 90 分钟时分钟被钉死、秒针却自循环，出现 `90:59→90:00` 反复不进位。
> 现统一为**同源 K(t)**，两者进位严格同步彻底修复。

### 补时显示
进入常规结束分钟（45/90/120）后**不再钉死**，比赛进入补时会自然继续上升（`90:59 → 91:00 → …`），
直到 API 返回 HT/FT（`!running`）时由上层 `LiveMinute` 定格为「中场休息 / 完场」。

### 暂停态不走秒
中场休息(HT) / 加时中场(BT) / 中断(SUSP/INT) / 点球大战(P) 这些状态下，`running=false`，
钟定格只显示中文状态，不逐秒跳。映射见 `lib/data.ts` 的 `liveClockFromStatus()`。

### 回退
若库里没有真实字段（旧数据 / 免费档不返回分钟），`LiveMinute` 回退到
`LiveMinuteEstimate`，按开球墙上时钟估算，显示"约 xx'"。

## 四、API 状态短码 → 中文（status_short）

`liveClockFromStatus()` 的完整映射（league=1 世界杯实测覆盖）：

| 短码 | 含义 | 中文显示 | 走秒？ |
|------|------|----------|--------|
| `NS`   | Not Started | 未开赛 | 否 |
| `1H`   | First Half | 上半场 | ✅ |
| `HT`   | Halftime | 中场休息 | 否（定格 45） |
| `2H`   | Second Half | 下半场 | ✅ |
| `ET`   | Extra Time | 加时赛 | ✅ |
| `BT`   | Break Time（加时两节之间） | 加时中场 | 否 |
| `P`    | Penalty In Progress | 点球大战 | 否 |
| `SUSP` | Suspended | 比赛中断 | 否 |
| `INT`  | Interrupted | 暂停中 | 否 |
| `FT`   | Match Finished（常规时间） | 完场 | 否 |
| `AET`  | After Extra Time | 加时完场 | 否 |
| `PEN`  | After Penalties | 点球完场 | 否 |
| `PST`  | Postponed | 延期 | 否 |
| `CANC` | Cancelled | 取消 | 否 |
| `ABD`  | Abandoned | 腰斩 | 否 |

未知短码：有 `elapsed` 当进行中（显示"进行中"并走秒），否则当未开赛。

## 五、自适应拉取频率（poller）

定义在 `scripts/poller.mjs` 顶部（可用环境变量覆盖）：

| 场景 | 频率 | 常量 | 这一轮干什么 |
|------|------|------|--------------|
| 有进行中(live)的场 | **20 秒/轮** | `FREQ_LIVE` / `POLL_LIVE_MS` | 比分 + 时间(分钟/补时) + 事件(仅有进展时拉) + **技术统计(每场按 `POLL_STATS_MS` 默认 20s 节流)** |
| 有 3h 内将开赛的场 | **3 分钟/轮** | `FREQ_PRE` / `POLL_PRE_MS` | 探状态 + 补阵容/首发/裁判 |
| 仅有 >3h 的赛前场 | 3 分钟/轮 | `FREQ_PRE` | 探状态 + 低频探首发 |
| 今日无 upcoming/live | **30 分钟/轮** | `FREQ_IDLE` / `POLL_IDLE_MS` | 收尾 / 等明天 |

- 临赛前窗口 `PRE_WINDOW_MS = 3h`。
- 每轮用**批量** `/fixtures?date=昨天&今天` 一次拿回当天所有场的状态（覆盖跨 UTC 边界的场，
  如法国-塞内加尔被 API 归到前一天）。
- **事件**：每个进行中场仅在"进展指纹"变化时才拉（指纹 = `比分|elapsed|extra|short`）。
- **技术统计**：进行中场调 `/fixtures/statistics?fixture=` 拉控球率/射门/射正/xG/角球/犯规/传球/越位写 `match_stats`，
  每场按 `POLL_STATS_MS`（默认 20s）节流强制重拉（`pullStats(..., force=true)`）；完赛时再拉一次最终版（含 xG）。
  统计是**按场**独立端点（不像比分能批量），所以每个直播场每轮多 1 次请求——这是赛中额度的主要增量。
- **首发阵容**：每场只拉一次（赛前 `predicted`，开赛后 `confirmed`）。
- **裁判**：从 fixture 里直接读，0 额外请求。
- **开球时间**：用 API 真实时间戳写 `kickoff_ms`（修了早先 8 小时偏差的 bug）。

## 五点五、夺冠榜实时引擎（随真实赛果更新）

夺冠概率榜不再是写死的静态表，而是「**已完赛锁定 + 蒙特卡洛模拟 + 仅赛果变化时重算**」的实时快照。

### 链路
```
真实赛果(RESULTS 静态 ∪ live_scores 已完赛)
   └─►(poller 每轮 tick 末尾，hash 变才跑)─► 2 万次锦标赛蒙特卡洛 ─► champion_sim 表
        └─►(服务端读库)─► app/champions 页 / layout 注入 / /api/live
             └─►(客户端轮询)─► useChampionRace() ─► 首页榜单等组件
```

### 各环节
| 环节 | 文件 | 作用 |
|------|------|------|
| 结构快照 | `scripts/lib/wc-structure.json` | 由 `scripts/gen-wc-structure.mts`（tsx 一次性）从 `lib/data.ts` 导出：球队排名/分组、72 场小组赛+海拔、32 场淘汰赛席位、东道主、静态赛果 |
| 模拟器核心 | `scripts/lib/champion-sim-core.mjs` | 复用 `lib/prediction-v2.ts` 的 `lambdasFromRanks` λ 引擎 + 泊松抽样；模拟小组赛→最佳第三名→淘汰赛全流程，输出 6 档概率 |
| 运行器 | `scripts/run-champion-sim.mjs` | 命令行手动重算（`node scripts/run-champion-sim.mjs [--force]`）；读已完赛→算 hash→变了才模拟写库 |
| poller 触发 | `scripts/poller.mjs` 的 `maybeRunChampionSim()` | 每轮 tick 末尾调用；进程内同步跑（约 2 秒），hash 未变零成本跳过 |
| 库表 | `lib/db/schema.ts` `lib/db/client.ts` | `champion_sim`（每队一行）+ `champion_sim_meta`（`results_hash` 去重判据） |
| 读库 | `lib/db/champion-sim.ts` | `readChampionSim()` 取快照；`getChampionRace()` 组装排序榜单，DB 空回退静态 |
| 组装纯函数 | `lib/data.ts` 的 `buildChampionRace(sim?)` | 由一份概率表组装排序榜；服务端/客户端共用 |
| 客户端 hook | `components/live-provider.tsx` 的 `useChampionRace(count?)` | 经 `/api/live` 的 `championRace` 字段实时更新，回退静态 `CHAMPION_RACE` |

### 关键设计
- **只锁定已完赛**：已踢的场用真实比分代入（积分/净胜球确定），未踢的按 v6 Elo λ 泊松抽样。
  进行中的比赛本期**仍按赛前 Elo**（不接 `predictInPlay`），结束写库后自然转为确定结果。
- **仅赛果变化时重算**：`results_hash = 排序后的「key:比分」串` 的哈希。poller 每轮算一次 hash，
  与 `champion_sim_meta` 比对，相同则跳过——所以赛中滚动（比分未定）**不会**触发，避免无谓重算。
- **朝向校正**：`buildFinished()` 把 live 的主客顺序对齐到赛程编排的 `home-away`，并只纳入小组赛场次
  （淘汰赛结果由模拟推演，不作输入约束）。运行器与 poller 同口径。
- **守恒自检**：48 队夺冠率总和≈100、出线率总和≈3200（32 队晋级）、决赛率≈200。可作回归断言。
- **东道主加成**：美/墨/加计入主场优势 + 墨西哥城/瓜达拉哈拉高原 `alt`，故其概率可能略高于纯实力排序。

### 重新生成结构快照（赛制/分组变动时）
```
npx tsx scripts/gen-wc-structure.mts     # 重新从 lib/data.ts 导出 wc-structure.json
node scripts/run-champion-sim.mjs --force  # 强制重算一次写库
```

## 六、额度够不够（Pro 档 7500 次/天）

按"每天 4 场、互不同时"估算：

- 批量状态：1 次/轮 × 全天 ≈ 几百次。
- 每场赛中 20s × 约 120 分钟 ≈ 360 轮，但事件仅在有进展时拉，比分/时间随批量一起回，
  单场实际新增请求远小于 360。
- 4 场合计实测约 **2000–3000 次/天**，占额度 **27%–40%**，留足余量。
- 想更实时：把 `POLL_LIVE_MS` 调到 10000（10s）仍在安全区（单场翻倍，4 场约 79%，偏紧）。
  默认保留 20s；如需关键时刻更密，可在 poller 里加"临近半场结束/点球"时段降到 10s。

## 七点五、手动补事件（events.source = api / manual）

API-Football 的 `/fixtures/events` **不收录被 VAR 取消的进球/点球、VAR check 等事件**
（实测法-塞第 60 分钟姆巴佩点球被 VAR 取消，API 始终不返回；SofaScore 有，是因为它用更全的源）。
为此 `events` 表加了 `source` 列：

- `source='api'`：poller 从 API 拉来的，**每轮整体删了重写**（会被覆盖更新）。
- `source='manual'`：手动录入的，**poller 重写时只删 `api` 行、保留 `manual` 行**，
  合并后按 `(minute, extra)` 时间顺序重排 `seq`，所以手动事件会穿插到正确分钟且不被冲掉。

**手动补一条事件**：编辑 `scripts/_add-manual-event.mjs` 里的 `matchKey / fid / manual` 字段后运行：
```bash
node scripts/_add-manual-event.mjs
```
`type` 用 API 的英文枚举（`Goal/Card/subst/Var`），前端 `eventLabel()` 会映射成中文
（`Var→VAR`，并显示警告三角图标）。脚本对"同分钟同类型"做了去重，重复运行不会叠加。

> ⚠️ 改 poller 的事件逻辑后**必须重启 poller 进程**（守护进程跑的是旧代码，不重启不生效）。
> 重启：`Stop-Process` 掉旧 node，再 `POLLER_ENABLED=1 API_FOOTBALL_KEY=... node scripts/poller.mjs`。

## 八、夺冠榜实时模拟（champion_sim）

夺冠概率榜**不再是静态表**，而是随真实赛果实时更新的蒙特卡洛模拟快照。

### 数据链路
```
poller（每轮 tick 末尾）
  └─ 读 live_scores 已完赛(status=finished) + 静态 RESULTS 打底 → 规范化 finished 集合
     └─ 算 resultsHash，与 champion_sim_meta.results_hash 比对
        ├─ 相同 → 跳过（零成本）
        └─ 变化 → simulate(finished, CHAMP_N_SIMS) 跑 2 万次锦标赛
                  └─ 写 champion_sim（48 行）+ champion_sim_meta（hash/次数/时间）

读取：
  · 服务端：getChampionRace()（lib/db/champion-sim.ts）读库 → buildChampionRace，DB 空回退静态 CHAMPION_RACE
      - app/champions/page.tsx（force-dynamic）
      - app/layout.tsx 注入 LiveProvider 初值（首屏即实时）
  · 客户端：/api/live 带 championRace → LiveProvider 轮询 → useChampionRace() hook
      - components/champion-race.tsx（首页榜单）
```

### 模拟方法（「只锁定已完赛」口径）
- **已完赛**：用真实比分锁死（含 live_scores 实时结果 + 静态 RESULTS）。
- **进行中**：本期**仍按赛前 Elo 抽样**（未接 in-play 实时态势，刻意从简）。
- **未开赛**：v6 Elo 多因子引擎的 λ 泊松抽样。
- 每组前 2 + 最佳 8 个第三名出线；淘汰赛按席位标签推演至决赛；点球按 λ 胜率简化。
- 核心：`scripts/lib/champion-sim-core.mjs`；结构数据快照：`scripts/lib/wc-structure.json`
  （由 `scripts/gen-wc-structure.mts` 经 `npx tsx` 从 `lib/data.ts` 一次性生成）。

### 手动重算
```bash
node scripts/run-champion-sim.mjs          # 赛果变化才算
node scripts/run-champion-sim.mjs --force  # 强制重算
```

> ⚠️ 改 `champion-sim-core.mjs` 或 poller 集成后**同样要重启 poller**。
> 改了 `lib/data.ts` 的赛程/分组/排名后，需重跑 `npx tsx scripts/gen-wc-structure.mts` 重新生成结构快照。

## 七点六、主教练绑定到球队（赛前不等 API 就有）

赛前 API 还没公布阵容时，教练原本也是空的（教练随 `/fixtures/lineups` 一起来）。
为此把 **48 队主帅绑定到球队代码**：`lib/db/team-coaches.ts` 的 `TEAM_COACHES`（队代码 → 中文名+战术风格）。

**优先级（在 `app/match/[id]/page.tsx` 实现）**：
1. **API/库里有实际阵容教练** → 用它（最准，含临时代帅）；
2. 否则 → 用 `TEAM_COACHES` 的**绑定值兜底**（赛前即显示）。

所以比赛详情页的「主教练信息」区块：赛前就显示绑定主帅，开赛 API 拉到真实阵容后自动切换成 API 的值。
**换帅时手动改 `team-coaches.ts` 即可**，不依赖 API 时机。

> 注意：`coaches` 这个 realData 标记现在只要"两边都拿得到教练（API 或绑定）"就为 true，
> 区块即展示；不再要求必须有 DB 阵容。

## 八、常见疑问速查

- **"比赛事件少了一条 VAR/被取消的进球"** → API-Football 源头就没有这条，不是 bug。
  用 `scripts/_add-manual-event.mjs` 手动补，会永久保留（见上「手动补事件」）。
- **"手动补的事件下一轮就没了"** → 改了 poller 代码但**没重启 poller 进程**，旧代码仍整场删重写。
- **"比赛开始了库里还是空的"** → poller 没在跑，或 key 没设。先 `pm2 logs` / 看 `poller.log`。
- **"分钟在 45↔23 之间跳"** → 旧估算逻辑的现象。现在有真实 `elapsed` 后走精确状态机，
  无真实字段才回退估算并显示"约"。
- **"开球时间差 8 小时"** → 早先 bug，poller 现在用 API 真实时间戳写 `kickoff_ms`，已修。
- **"姆巴佩/亚马尔首发状态"** → 来自 `lib/star-players.json` 球星表 + 阵容数据，
  详见预测/球星可用性相关代码，不在本文档范围。
